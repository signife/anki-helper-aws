# ANKI-HELPER 요구사항 정의서

## 1. 문서 목적

본 문서는 AWS 기반 AI 어휘 카드 생성 서비스 **ANKI-HELPER**가 제공해야 하는 기능적 요구사항과 비기능적 요구사항을 정의한다.

프로젝트의 목적, 사용자 구분 및 전체 서비스 범위는 `01-spec-overview-anki-helper.md`에서 정의하고, 기술·보안·AWS 서비스·배포 환경 관련 제약사항은 `03-constraints-anki-helper.md`에서 정의한다.

본 문서는 **회원가입과 로그인을 MVP에 포함하는 것**을 전제로 한다.

### 2.2 중요도 기준

| 중요도 | 설명 |
|---|---|
| 상 | MVP 구현에 반드시 필요한 요구사항 |
| 중 | 서비스 완성도와 안정성을 위해 필요한 요구사항 |
| 하 | 향후 확장 가능한 선택 요구사항 |

### 2.3 요구사항 상태

| 상태 | 설명 |
|---|---|
| 미구현 | 아직 개발하지 않은 상태 |
| 구현 중 | 현재 개발 중인 상태 |
| 구현 완료 | 기능 구현이 완료된 상태 |
| 테스트 완료 | 관련 테스트까지 완료된 상태 |
| 보류 | 정책 또는 외부 서비스 확인이 필요한 상태 |

# 5. 이메일 인증 요구사항

| 요구사항 ID | 요구사항명 | 상세 내용 | 중요도 |
|---|---|---|---|

| REQ_VERIFY_002 | 인증 코드 입력 | 사용자는 웹 화면에서 이메일 인증 코드를 입력할 수 있어야 한다. | 상 |
| REQ_VERIFY_003 | 인증 완료 | 올바른 인증 코드가 확인되면 계정을 로그인 가능한 상태로 전환해야 한다. | 상 |
| REQ_VERIFY_004 | 인증 전 로그인 제한 | 이메일 인증을 완료하지 않은 계정은 회원 전용 기능을 사용할 수 없어야 한다. | 상 |
| REQ_VERIFY_005 | 인증 코드 재전송 | 인증 코드를 받지 못한 사용자는 재전송을 요청할 수 있어야 한다. | 상 |
| REQ_VERIFY_006 | 재전송 대기시간 | 인증 코드 재전송을 연속으로 요청할 수 없도록 대기시간을 적용해야 한다. | 상 |
| REQ_VERIFY_007 | 재전송 횟수 제한 | 일정 시간 내 인증 코드 재전송 횟수를 제한해야 한다. | 상 |
| REQ_VERIFY_008 | 만료 코드 처리 | 만료된 인증 코드 입력 시 만료 안내와 재전송 기능을 제공해야 한다. | 상 |
| REQ_VERIFY_009 | 잘못된 코드 처리 | 올바르지 않은 인증 코드 입력 시 남은 재시도 또는 재전송 방법을 안내해야 한다. | 상 |
| REQ_VERIFY_010 | 중복 인증 처리 | 이미 인증된 계정에 인증을 다시 시도하면 인증 완료 상태임을 안내해야 한다. | 중 |
| REQ_VERIFY_011 | 이메일 마스킹 | 인증 화면에서는 이메일 주소 일부를 마스킹하여 표시해야 한다. | 중 |
| REQ_VERIFY_012 | 인증정보 로그 제외 | 인증 코드 원문을 애플리케이션 로그에 기록해서는 안 된다. | 상 |

### 이메일 인증 흐름

```text
회원가입 요청
    ↓
Cognito 사용자 생성
    ↓
이메일 인증 코드 발송
    ↓
사용자 인증 코드 입력
    ↓
Cognito 인증 코드 검증
    ↓
계정 확인 완료
    ↓
로그인 가능
```

# 6. 로그인·로그아웃·토큰 요구사항

| REQ_AUTH_003 | 인증 완료 확인 | 이메일 인증이 완료된 계정만 정상 로그인할 수 있어야 한다. | 상 |
| REQ_AUTH_004 | 공통 실패 메시지 | 이메일 또는 비밀번호 오류 시 계정 존재 여부를 과도하게 노출하지 않아야 한다. | 상 |
| REQ_AUTH_005 | 미인증 계정 안내 | 미인증 계정 로그인 시 인증 필요 안내와 코드 재전송 기능을 제공해야 한다. | 상 |
| REQ_AUTH_006 | 로그인 성공 | 로그인 성공 시 Cognito가 발급한 유효한 토큰을 이용해 회원 전용 API를 호출할 수 있어야 한다. | 상 |
| REQ_AUTH_007 | API 토큰 전달 | 회원 전용 API 요청에는 유효한 Access Token 또는 정책상 지정한 JWT를 포함해야 한다. | 상 |
| REQ_AUTH_008 | API Gateway 검증 | API Gateway는 Lambda 실행 전에 Cognito JWT의 서명, 발급자 및 만료 여부를 검증해야 한다. | 상 |
| REQ_AUTH_009 | 사용자 식별 | Lambda는 검증된 토큰의 `sub` 값을 사용자 식별값으로 사용해야 한다. | 상 |
| REQ_AUTH_010 | 타인 ID 금지 | 클라이언트가 전송한 임의의 사용자 ID를 데이터 소유권 판정에 사용해서는 안 된다. | 상 |
| REQ_AUTH_011 | 토큰 만료 처리 | 토큰이 만료된 경우 갱신 가능한 범위에서는 갱신하고, 불가능하면 다시 로그인하도록 안내해야 한다. | 상 |
| REQ_AUTH_012 | 로그아웃 | 로그인 사용자는 로그아웃할 수 있어야 한다. | 상 |
| REQ_AUTH_013 | 로컬 인증정보 제거 | 로그아웃 시 브라우저에 보관된 인증 관련 정보를 제거해야 한다. | 상 |
| REQ_AUTH_014 | 보호 화면 이동 | 로그아웃 후 회원 전용 화면에 접근하면 로그인 화면으로 이동해야 한다. | 상 |
| REQ_AUTH_015 | 로그인 상태 표시 | 화면에서 현재 로그인 사용자와 로그아웃 기능을 확인할 수 있어야 한다. | 중 |
| REQ_AUTH_016 | 반복 로그인 제한 | 반복적인 로그인 실패는 Cognito 정책 또는 추가 제한 정책에 따라 차단해야 한다. | 상 |
| REQ_AUTH_017 | 토큰 로그 제외 | ID Token, Access Token 및 Refresh Token 원문을 CloudWatch 로그에 기록해서는 안 된다. | 상 |

# 29. MVP 사용자 흐름

## 29.1 신규 사용자 흐름

```text
웹사이트 접속
    ↓
회원가입
    ↓
이메일 인증 코드 입력
    ↓
로그인
    ↓
기본 모국어·카드 모드·음성 설정
    ↓
모바일 또는 PC에서 일본어 단어 입력
    ↓
Bedrock 카드 생성
    ↓
카드 미리보기·수정
    ↓
개인 수집함에 READY 상태 저장
```

## 29.2 모바일 수집 후 PC Anki 추가 흐름

```text
모바일에서 로그인
    ↓
단어 입력 및 카드 생성
    ↓
DynamoDB 개인 수집함에 저장
    ↓
집의 PC에서 같은 계정으로 로그인
    ↓
Anki Desktop 실행
    ↓
수집함의 READY 카드 선택
    ↓
Polly 음성 생성
    ↓
브라우저가 localhost:8765의 AnkiConnect 호출
    ↓
Anki 미디어 저장
    ↓
Anki 카드 추가
    ↓
성공 카드 상태를 ADDED로 변경
```

## 29.3 일괄 추가 흐름

```text
PC에서 READY 카드 여러 개 선택
    ↓
AnkiConnect 연결 및 노트 타입 확인
    ↓
카드별 Polly 음성 생성
    ↓
카드별 미디어 저장
    ↓
카드별 Anki 추가
    ↓
성공·실패 결과 집계
    ↓
성공 카드만 ADDED 상태 반영
    ↓
실패 카드 재시도 제공
```

# 32. 완료 기준

MVP는 다음 조건을 충족하면 주요 요구사항을 완료한 것으로 판단한다.

1. 사용자가 이메일로 가입하고 인증한 뒤 로그인할 수 있다.
2. 인증된 사용자만 Bedrock과 Polly 기반 기능을 사용할 수 있다.
3. 사용자가 모바일에서 생성한 카드를 개인 수집함에 저장할 수 있다.
4. 동일 사용자가 PC에서 로그인하여 저장한 카드를 조회할 수 있다.
5. Bedrock이 정의된 카드 데이터 구조를 생성하고 Lambda가 검증한다.
6. 사용자가 생성 결과를 확인하고 수정할 수 있다.
7. DynamoDB에서 사용자별 카드와 설정이 분리된다.
8. 사용량 제한으로 비정상적인 Bedrock·Polly 호출을 제어할 수 있다.
9. Polly가 단어, 표현 및 예문 음성을 생성해 브라우저에 반환한다.
10. 브라우저가 로컬 AnkiConnect에 미디어와 카드를 저장한다.
11. 필요한 Anki 노트 타입, 18개 필드, 템플릿 및 CSS를 자동 구성할 수 있다.
12. 여러 카드를 선택해 일괄 추가하고 카드별 결과를 확인할 수 있다.
13. 성공한 카드는 `ADDED`, 실패한 카드는 재시도 가능한 상태로 관리된다.
14. React 정적 파일은 S3와 CloudFront를 통해 배포된다.
15. AWS 인프라를 Terraform으로 재현할 수 있다.
16. GitHub Actions가 테스트, 빌드 및 배포를 자동화한다.
17. CloudWatch에서 주요 오류, 처리시간, 사용량 및 비용 이상을 확인할 수 있다.
18. 비밀번호, JWT, 인증 코드, AWS 키 및 전체 음성 데이터가 로그나 프런트 코드에 노출되지 않는다.
19. 계정 탈퇴 시 Cognito 계정과 사용자별 DynamoDB 데이터를 삭제할 수 있다.
20. EC2, ALB, RDS, Redis 및 Jenkins 서버 없이 서버리스 구조로 운영할 수 있다.
